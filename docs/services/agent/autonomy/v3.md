# Autonomy v3 — Reactive agenda + user-programmable kinds

v3 turns the autonomy engine from "two scheduled sweeps plus nightly memory
consolidation" into a general-purpose reactive layer. Users can author
their own agenda items through the `/autonomy` dashboard, drive them from
Home Assistant webhooks or live MQTT, and have the engine honor
quiet-hours, per-item rate limits, and a live WebSocket run feed — all
inside the existing `observe` / `notify` tool gating.

This document is the user- and operator-facing complement to the design
spec at `/home/matt/.claude/plans/mellow-sauteeing-pond.md`. See
[README.md](README.md) for v1 (briefing / anomaly sweep) and
[memory/README.md](memory/README.md) for v2 (L1–L4 memory tiers).

## What v3 adds

Three new agenda kinds:

- **`reminder`** — deterministic notify, no LLM. Config:
  `{title, body, channel, to?, one_shot?, quiet_hours?}`. When `one_shot` is
  true, the engine **hard-deletes** the agenda row after the first successful
  fire so it disappears from the dashboard's Agenda Items list. The audit
  trail is preserved: the corresponding `autonomy_runs` row stays with its
  `agenda_item_id` set to NULL via the schema's `ON DELETE SET NULL` FK.
  See [tools/reminder.md](../tools/reminder.md) for the LLM-facing scheduling
  tools that produce these items.
- **`watch`** — reactive trigger-driven notify. Template-renders a
  `body_template` against the trigger event payload (plus `_topic`, `_name`,
  `_source`, `_ts`), optionally gates on a Home Assistant entity state
  (`condition.entity_id` + `condition.state` + `condition.min_duration_sec`),
  and routes through the same anomaly-style notification path so the
  existing per-signature cooldown keeps watchers quiet during bursts.
- **`routine`** — goal-oriented LLM turn. Builds a fresh `AutonomousTurn`
  (same isolated orchestrator used by v1 briefing / anomaly) with an
  optional per-item `tools_override` that must be a *subset* of the current
  tier's allow-list. Delivers the final message via `deliver.channel`
  (`signal` / `ha_push`).

Two new reactive event sources:

- **HA webhook intake** — `POST /api/autonomy/webhook/{name}`. The `{name}`
  segment is the shared secret (home-lab trust model, no auth). The body is
  matched against every enabled `trigger_spec.source='webhook'` item
  targeting `{name}`; each match fires its handler. Bring your own HA
  automation to POST into this endpoint on state change.
- **Live MQTT subscriber** — a paho-mqtt client on a background thread is
  wired to the existing mosquitto broker. Subscriptions are computed from
  enabled `trigger_spec.source='mqtt'` items; CRUD writes raise an
  `asyncio.Event` that diff-subscribes the listener without reconnecting.
  Topic wildcards support the single-level `+` (no `#`). Payload matching
  is key-subset — every key you list in `trigger_spec.match.payload` must
  equal the incoming payload's value for that key; unlisted keys are
  ignored.

Plus:

- **Quiet hours** (`config.quiet_hours` with `policy='defer'|'drop'`) —
  `defer` inserts a `status='scheduled'` row with `scheduled_for =
  next_end_of_quiet`; the dispatcher sweeps these back up on each tick and
  fires as soon as the window ends. `drop` records
  `status='skipped_quiet_hours'` and walks away.
- **Per-item event rate limit** — leaky-bucket limiter keyed on item id,
  parsed from shorthand (`N/sec`, `N/min`, `N/hr`). Applies to reactive
  triggers only; cron dispatch is still gated by the global hourly limit.
- **Live run feed** — `WS /ws/autonomy/runs` streams every newly-inserted
  `autonomy_runs` row via a Postgres `LISTEN` on channel `autonomy_runs_ch`.
  A trigger on `autonomy_runs` emits the new row id on each `INSERT`; the
  WS endpoint acquires a dedicated pool connection, primes the client with
  the most recent 25 runs, then pushes one frame per row as they land.
- **Dashboard** — `/autonomy` in the agent's SvelteKit frontend: engine
  stats, agenda CRUD (form with radio-selected kind + live JSON preview),
  live run feed, reactive-source health cards, filterable run history.

## Flags and new environment variables

v3 ships flagged off — ensure-schema migrates idempotently, tables grow new
columns, but the MQTT listener and webhook intake stay dormant until you
opt in.

```bash
# Core flags (default: off). Flip once you have agenda items in place.
AUTONOMY_WEBHOOK_ENABLED=false
AUTONOMY_MQTT_ENABLED=false

# MQTT subscriber wiring. MQTT_BROKER / MQTT_PORT are reused from the
# existing mqtt_tools config.
AUTONOMY_MQTT_CLIENT_ID="selene-autonomy"
AUTONOMY_MQTT_RECONNECT_MAX_SEC=60

# Default quiet-hours window applied when an item omits its own spec.
# Leave empty to disable global default.
AUTONOMY_DEFAULT_QUIET_START=""         # "22:00"
AUTONOMY_DEFAULT_QUIET_END=""           # "07:00"
AUTONOMY_DEFAULT_QUIET_POLICY="defer"   # "defer" | "drop"

# Default per-item event-trigger rate limit. Reactive triggers only.
AUTONOMY_DEFAULT_EVENT_RATE_LIMIT="10/min"
```

All eight live together with the v1/v2 autonomy env block in `.env.tmpl`,
`shared/configs/shared_config.py`, and `services/agent/selene_agent/utils/config.py`.

## Schema additions

Applied idempotently at startup by `autonomy.db.ensure_schema()`:

- `agenda_items.trigger_spec jsonb` — `{source: 'mqtt'|'webhook', match: {...}}`.
  Nullable; `schedule_cron IS NOT NULL OR trigger_spec IS NOT NULL` is a
  table-level `CHECK`.
- `agenda_items.name text` — optional operator-facing label.
- `agenda_items.schedule_cron` loses its `NOT NULL` constraint (reactive
  items are cron-less).
- `autonomy_runs.scheduled_for timestamptz` — set when a run is deferred
  past quiet hours.
- `autonomy_runs.trigger_source text` — `cron` / `mqtt` / `webhook` /
  `manual` / `deferred`.
- `autonomy_runs.trigger_event jsonb` — raw event payload for reactive
  runs; `null` for cron.
- New trigger `autonomy_runs_notify` + channel `autonomy_runs_ch` for the
  live WS feed.

## REST / WS surface

All under `/api/autonomy/*` (agent port 6002, no auth — same convention as
the rest of the agent API).

| Method | Endpoint | Purpose |
|--------|----------|---------|
| `GET`  | `/status` | Engine state, runs-in-last-hour, MQTT health, deferred queue depth, next-due preview, active `timezone` (the server's `CURRENT_TIMEZONE`, consumed by the dashboard to label cron inputs without hardcoding) |
| `POST` | `/pause` / `/resume` | Runtime kill switch |
| `GET`  | `/items` | List every agenda item |
| `POST` | `/items` | Create a new agenda item |
| `PATCH` | `/items/{id}` | Partial update; re-validates cron + trigger_spec; raises MQTT refresh |
| `DELETE` | `/items/{id}` | Delete |
| `GET`  | `/runs?limit=50&kind=&status=&trigger_source=&include_messages=0` | Filterable run history |
| `POST` | `/trigger/{id}?bypass_quiet=true` | Manual fire; `bypass_quiet` ignores the quiet-hours gate |
| `POST` | `/webhook/{name}` | HA webhook intake; matches body against `trigger_spec.source='webhook'` items |
| `GET`  | `/events/summary` | MQTT subscribed topics + webhook-item names + runs-by-source over last 24h + deferred queue depth |

`WS /ws/autonomy/runs` — primes with the last 25 runs, then streams one
JSON frame `{type: 'run', run: {...}}` per newly-inserted row.

The agent's `/health` gains `mqtt_connected`, `subscribed_topics`, and
`deferred_runs_pending` inside the `autonomy` block.

## Config shapes

### `reminder`

```json
{
  "kind": "reminder",
  "name": "Laundry 17:00",
  "schedule_cron": "0 17 * * *",
  "autonomy_level": "notify",
  "config": {
    "title": "Laundry",
    "body": "Start a load — weather window opens in 2h.",
    "channel": "ha_push",
    "to": "mobile_app_pixel_8",
    "one_shot": false,
    "quiet_hours": { "start": "22:00", "end": "07:00", "policy": "defer" }
  }
}
```

### `watch` (MQTT-triggered)

```json
{
  "kind": "watch",
  "name": "Front door after midnight",
  "trigger_spec": {
    "source": "mqtt",
    "match": { "topic": "home/door/front/state", "payload": { "state": "open" } }
  },
  "autonomy_level": "notify",
  "config": {
    "body_template": "Front door opened at {_ts}",
    "channel": "ha_push",
    "severity": "warn",
    "quiet_hours": { "start": "00:00", "end": "06:00", "policy": "defer" },
    "event_rate_limit": "3/min",
    "condition": { "entity_id": "binary_sensor.front_door", "state": "on", "min_duration_sec": 0 }
  }
}
```

### `routine`

```json
{
  "kind": "routine",
  "name": "Sunday weekly recap",
  "schedule_cron": "0 18 * * 0",
  "autonomy_level": "notify",
  "config": {
    "prompt": "Summarize this week's memories and suggest three focus areas for next week.",
    "tools_override": ["search_memories", "ha_send_notification"],
    "deliver": { "channel": "signal", "to": "+15551234567" }
  }
}
```

`tools_override` **must be a subset** of the current `autonomy_level` tier's
allow-list; the engine raises on mismatch (surfaced as a failed run with a
clear error).

## Trigger matching rules

`autonomy/trigger_match.py` is the single source of truth. An incoming event
matches an item's `trigger_spec` iff:

1. `trigger_spec.source == event.source` (strict equality).
2. **MQTT**: `trigger_spec.match.topic` matches the incoming topic, with `+`
   acting as a single-level wildcard (no `#`). Matching is literal otherwise.
3. **Webhook**: `trigger_spec.match.name == event.name` exactly.
4. `trigger_spec.match.payload` is a **recursive key subset** — every key
   listed must exist in the incoming payload with an equal value; payloads
   may contain additional keys without losing the match.

If the trigger_spec resolves to a non-match, the engine still inserts a
`status='skipped_trigger_mismatch'` run row for auditability.

## Quiet hours

`config.quiet_hours = { start, end, policy }`, timezone-aware via
`CURRENT_TIMEZONE`. Cross-midnight windows (`22:00` → `07:00`) work. When
quiet hours suppress a run:

- `policy='defer'` — insert `status='scheduled'` + `scheduled_for =
  next_end_of_quiet`. The dispatcher sweeps every scheduled row that has
  become due on each tick and fires it with `trigger_source='deferred'`.
  Claiming uses `DELETE ... RETURNING` so there's no lingering `in_flight`
  state — the subsequent fire creates a fresh `autonomy_runs` row.
- `policy='drop'` — insert `status='skipped_quiet_hours'` and walk away.

`POST /api/autonomy/trigger/{id}?bypass_quiet=true` skips this gate.

If `config.quiet_hours` is omitted, the engine falls back to the
`AUTONOMY_DEFAULT_QUIET_*` env. Malformed specs fail open (treated as no
quiet hours) rather than silently blocking the item.

## Event rate limiting

`config.event_rate_limit = "N/sec" | "N/min" | "N/hr"`. Applied **only** to
reactive triggers (`mqtt` / `webhook`). Implementation is an in-memory
leaky bucket keyed on `item_id`; overflow is recorded as
`status='skipped_rate_limit'`. The global hourly `AUTONOMY_MAX_RUNS_PER_HOUR`
cap still applies on top.

## Dashboard

`/autonomy` (agent port 6002):

- **Header strip** — engine status, pause / resume, "New item" button.
- **Engine stats** — runs / hr with cap, deferred queue depth, MQTT
  connection + subscribed topic count, live WS status.
- **Agenda items** — table with per-row Run / Edit / Delete actions, an
  inline enabled toggle, and trigger description.
- **Live run feed** — WS-backed, last 50 runs.
- **Reactive sources** — currently subscribed MQTT topics + configured
  webhook paths + runs-by-source counts over the last 24h.
- **Run history** — filterable by kind / status / trigger_source.

The agenda editor (`AgendaForm.svelte`) renders kind-scoped fields and a
live JSON preview of the request body so you can see exactly what the API
will receive.


- `test_event_rate_limit.py` — burst + steady-state, `N/sec|min|hr` parsing.
- `test_reminder_handler.py`, `test_watch_handler.py`, `test_routine_handler.py`
  — handler return contract + `one_shot` disable + `tools_override` subset
  enforcement.
- `test_mqtt_listener.py` — subscription diffing, reconnect backoff.
- `test_autonomy_api.py` — CRUD round-trip, webhook fan-out.
- `test_deferred_runs.py` — quiet-hours `defer` → sweep → fire.

### Live integration (full stack up)

```bash
docker compose up -d
curl http://localhost:6002/api/autonomy/status
curl http://localhost:6002/health   # expect mqtt_connected=true once flag flipped
```

Representative hand checks once the flags are on:

1. Create a reminder via the dashboard; confirm `next_fire_at` in `/items`.
2. Create an MQTT watch on a topic you can publish to; `mosquitto_pub -h
   localhost -t <topic> -m '{...}'`; confirm a run row streams into the
   live feed.
3. Create a webhook watch; `curl -X POST -d '{"state":"open"}'
   http://localhost:6002/api/autonomy/webhook/<name>`; confirm matched +
   fired.
4. Build a quiet-hours window that covers now with `policy='defer'`;
   trigger the item; confirm `status='scheduled'`; walk the clock forward
   and confirm it fires.
5. Trigger repeatedly past the per-item rate limit and confirm
   `status='skipped_rate_limit'`.
6. Edit an MQTT item's topic via `PATCH`; confirm the listener
   diff-resubscribes (inspect `/events/summary`).
7. Delete an item; confirm its topic drops out of the subscribed set.
8. Disconnect the broker and confirm the listener reconnects with
   exponential backoff capped at `AUTONOMY_MQTT_RECONNECT_MAX_SEC`.

## Rollout

1. Schema migration runs on first start after deploy; zero-downtime
   because new columns are nullable and the `CHECK` allows pre-existing
   cron-only rows.
2. Leave `AUTONOMY_WEBHOOK_ENABLED` + `AUTONOMY_MQTT_ENABLED` at `false`.
   Create a handful of agenda items through the dashboard, trigger them
   manually, verify notifications land, inspect runs.
3. Flip the flags; soak for a week with real items (laundry reminder,
   front-door watch, Sunday recap routine).
4. Merge `clauding` → `main` after soak.

## Troubleshooting

- **WS `/ws/autonomy/runs` never connects** — check the Postgres trigger
  exists (`\d+ autonomy_runs` in psql). `ensure_schema()` creates it
  idempotently; if you nuked the trigger manually, restart the agent.
- **MQTT items created but `subscribed_topics` is empty** — confirm
  `AUTONOMY_MQTT_ENABLED=true`, the broker is reachable from the agent
  container (`MQTT_BROKER` / `MQTT_PORT`), and the item is `enabled=true`.
- **Deferred runs stuck pending** — the sweep runs on each dispatcher
  tick; if `AUTONOMY_ENABLED=false` or the engine is paused, scheduled
  rows stay put until the engine resumes.
- **`tools_override` blocks a routine** — the override must be a strict
  subset of the current tier's allow-list (`observe` / `notify`). Raise
  the autonomy_level or drop the over-reaching tool from the override.

## Deferred to v4

- `speak` / `act` autonomy tiers and a per-action permission model.
- LLM-judged watches (beyond the current deterministic template + HA
  state condition).
- Declarative routine DSL.
- Cross-item trigger chaining ("if watch X fires, dispatch routine Y").
- Smaller / cheaper triage model.
- Pattern learning from user-triggered routines.
