# Autonomy v4 ŌĆö Speaker announcements, actuation, LLM event triage

v4 extends v3's reactive autonomy layer with three capabilities the v3 design
deliberately deferred: TTS-rendered announcements on the Music Assistant
speaker mesh, a supervised **`act` tier** that can actuate devices through
explicit per-item permission grants, and an LLM-judged reactive kind
(`watch_llm`) for triage calls that don't reduce to a simple boolean
condition.

See [v3.md](v3.md) for the reactive agenda foundation (trigger_spec, webhook
intake, MQTT listener, quiet hours, WS live feed) and
[README.md](README.md) for v1 (briefing / anomaly sweep).

## What v4 adds

### New tier: `speak`

Routes through the same planner surface as `notify` (observe-tier read tools,
no actuators), but delivery goes through a `SpeakerNotifier` instead of
Signal or HA Push. The notifier:

1. Renders the reply with Kokoro TTS (`/v1/audio/speech`, configurable voice).
2. Parks the audio bytes in an in-process `AudioStore` keyed by a random
   `secrets.token_urlsafe(16)` token with a short TTL (default 10 min,
   single-fetch eviction).
3. Hands Music Assistant a URL of the form
   `{AGENT_BASE_URL}/api/tts/audio/{token}.mp3`.
4. Calls the new MCP tool `mass_play_announcement` ŌĆö which wraps
   `music_assistant_client`'s `Players.play_announcement(player_id, url,
   volume_level, pre_announce)` ŌĆö so MA handles volume ducking and resume.

Set `channel: "speaker"` on any `briefing`, `routine`, `reminder`, or
`watch_llm` item to route through the mesh. Config supports per-item
`speaker.device`, `speaker.voice`, `speaker.volume` (0.0ŌĆō1.0 float or 0ŌĆō100
int; normalised to MA's 0ŌĆō100 percent).

### New kind: `watch_llm`

Fires from an MQTT or webhook event (same `trigger_spec` matcher as `watch`),
but the engine hands the event + a bounded state gather to the LLM and asks
for a single JSON judgment:

```json
{"unusual": bool, "severity": "low|med|high|none", "summary": "...",
 "signature": "slug", "evidence": ["...", "..."]}
```

Use when "is this noteworthy?" can't be expressed as a template condition ŌĆö
a door opening during a time window where you *sometimes* expect the dog
walker, for instance. Severity floors (`cfg.severity_floor`) gate low-signal
judgments. Dedup reuses v1's per-signature cooldown path (same
`anomaly_sweep` plumbing ŌĆö the engine handles all three of `anomaly_sweep`,
`watch`, and `watch_llm` identically once a signature hash lands).

### New kind: `act` (two-phase, permission-gated)

`act` is the first autonomy kind that can change device state. To keep
supervision strict, the handler splits the LLM plan from the engine's
execution:

1. **Phase 1 ŌĆö plan.** The handler spins an `AutonomousTurn` at the
   `observe` tier, so the LLM can read state but **cannot actuate from
   inside the turn**. The system prompt asks for strict JSON:

   ```json
   {"steps": [{"tool": "ha_control_light", "args": {...},
               "rationale": "..."}],
    "reasoning": "..."}
   ```
2. **Phase 2 ŌĆö validate.** The engine iterates every proposed step and
   annotates an `action_audit` list: `pending` when the tool is inside the
   item's `action_allow_list`, `skipped_not_allowed` otherwise, `malformed`
   for missing fields. The audit is **never dropped** ŌĆö every proposed
   call is persisted for later review.
3. **Phase 3 ŌĆö execute.** If `require_confirmation=false`, the engine
   runs every `pending` step inline via the MCP manager and flips each to
   `executed` or `error`. If `require_confirmation=true` (default), the
   engine persists the run with `status='awaiting_confirmation'`, a random
   `confirmation_token`, the full action_audit, and sends a notification
   containing a deep-link of the form
   `{AGENT_BASE_URL}/autonomy?confirm={run_id}&token={token}`.

The confirm endpoint (`POST /api/autonomy/runs/{run_id}/confirm` with body
`{approved: bool, token: string}`) validates the token (constant-time
compare) and either cancels the run (`confirmation_denied`) or hands the
stored audit back to the act handler's `execute_approved` helper. The engine
also runs a confirmation-timeout sweep on each tick ŌĆö runs past their
`confirmation_timeout_sec` (default 300s) transition atomically to
`confirmation_timeout`.

**`act` ships flagged off.** Set `AUTONOMY_ACT_ENABLED=true` to enable the
tier; without it the handler short-circuits to an error row even if an
operator authors an `act` item.

Example `act` config:

```json
{
  "kind": "act",
  "autonomy_level": "act",
  "config": {
    "prompt": "Turn on the reading lamp in the living room.",
    "action_allow_list": ["ha_control_light", "ha_activate_scene"],
    "require_confirmation": true,
    "confirmation_timeout_sec": 300,
    "deliver": {"channel": "signal", "to": "+15551234567"}
  }
}
```

## Tier-to-tool matrix

`selene_agent/autonomy/tool_gating.py` now publishes four tiers:

| Tier     | Adds on top of previous tier                                   |
|----------|---------------------------------------------------------------|
| observe  | read-only HA state, weather, memory search, etc.              |
| notify   | + send_signal_message, ha_notify, persistent_notification     |
| speak    | = notify (same surface; delivery channel differs)             |
| act      | + ha_control_light, ha_control_switch, ha_control_climate,    |
|          |   ha_control_media_player, ha_activate_scene, ha_trigger_script, |
|          |   ha_execute_service, mass_play_media, mass_playback_control  |

`act`-tier items still enforce a per-item `action_allow_list`; the tier
grant is a *necessary* condition, the allow-list is the *sufficient* one.

## Schema additions (`autonomy_runs`)

All four columns are added `IF NOT EXISTS`, so re-running `ensure_schema` is
safe on upgraded clusters:

- `confirmation_token TEXT` ŌĆö random 24-byte token; never returned over
  list endpoints.
- `confirmation_prompt_id TEXT` ŌĆö reserved for future use (e.g. correlating
  inbound Signal replies).
- `confirmation_response TEXT` ŌĆö `approved` | `denied` | `timeout`.
- `action_audit JSONB` ŌĆö list of `{tool, args, rationale, outcome,
  result?, error?}` entries, one per proposed step.

Plus a partial index on `status='awaiting_confirmation'` so the timeout
sweep is cheap regardless of total run count.

New statuses: `awaiting_confirmation`, `confirmation_denied`,
`confirmation_timeout`.

## REST additions

- `POST /api/autonomy/runs/{run_id}/confirm` ŌĆö body
  `{approved: bool, token?: string}`; 404 if the run is missing, 409 if it
  is not in `awaiting_confirmation`, 403 on token mismatch.
- `GET /api/autonomy/runs/awaiting` ŌĆö unfiltered list of parked runs
  (token stripped).
- `GET /api/autonomy/runs/{run_id}` ŌĆö single-run fetch (token stripped);
  supports `?include_messages=1`.
- `GET /api/tts/audio/{token}.mp3` ŌĆö the public fetch endpoint MA hits;
  404 when the token is unknown, expired, or already consumed.
- `POST /api/tts/announce` ŌĆö body
  `{text, device, voice?, volume?}`. Convenience endpoint used by the
  dashboard "Speak to device" card: renders `text` through Kokoro, parks it
  in the audio store, and plays it on `device` via
  `mass_play_announcement`. Reuses `SpeakerNotifier`, so behavior matches
  the `speak` delivery channel used by autonomy items.
- `GET /api/tts/players` ŌĆö proxies `mass_list_players` and returns
  `{players: [{name, player_id, available, powered}]}` for the dashboard's
  device dropdown. MA player `display_name` values are surfaced as `name`
  to keep the frontend and `mass_play_announcement` keyed off the same
  field.

## New environment variables

```bash
# act tier kill switch (default false). Without this even an act-kind item
# falls through to an error row.
AUTONOMY_ACT_ENABLED=false

# Default per-item confirmation timeout when the item's config omits one.
AUTONOMY_ACT_DEFAULT_CONFIRMATION_TIMEOUT_SEC=300

# SpeakerNotifier defaults; overridable per item via cfg.speaker.*.
AUTONOMY_SPEAKER_DEFAULT_DEVICE=""        # Music Assistant player name
AUTONOMY_SPEAKER_DEFAULT_VOICE="af_heart" # Kokoro voice
AUTONOMY_SPEAKER_DEFAULT_VOLUME=0.5       # 0.0ŌĆō1.0 (normalised to 0ŌĆō100)
AUTONOMY_TTS_AUDIO_TTL_SEC=600            # AudioStore TTL

# Public base URL used in the confirmation deep-link and the MA-fetched
# audio URL. If AGENT_BASE_URL is unset, AGENT_INTERNAL_BASE_URL is used
# for MA audio fetches; if that is also unset the notifier falls back to
# http://agent:6002.
#
# The compose-network hostname (`http://agent:6002`) only works when Music
# Assistant shares the docker network. For the common deployment where MA
# runs on a separate host or VLAN, set AGENT_INTERNAL_BASE_URL to the docker
# host's LAN-routable address (e.g. http://10.0.0.134:6002) so MA can fetch
# the staged audio; otherwise MA plays its pre-announce chime and then
# silently fails the URL fetch.
AGENT_BASE_URL=""
AGENT_INTERNAL_BASE_URL="http://agent:6002"
```

## Dashboard

The TTS Playground (`/playgrounds/tts`) and Autonomy page (`/autonomy`) both
render a **Speak to device** card that lets an operator pick a Music
Assistant player, type a message, and play it immediately ŌĆö no agenda item
needed. The card is a single shared Svelte component
(`$lib/components/SpeakToDevice.svelte`) that calls `/api/tts/announce`
with the selected device/voice/volume. Device list comes from
`/api/tts/players`; when MA is unreachable or has no players the card
falls back to a free-text device input.

The Autonomy page additionally renders a **Pending confirmations** card
whenever one or more `act` runs are parked in `awaiting_confirmation`.
Each row shows the proposed `action_audit` (tool + rationale per step)
and offers approve/deny buttons. Deep-links of the form
`/autonomy?confirm={run_id}&token={token}` ŌĆö what the confirmation
notification points at ŌĆö scroll the card into view and use the URL token
when clicking Approve/Deny; the query params are cleared after a
response lands so a reload doesn't re-trigger the flow.

## Operational notes

- **`awaiting_confirmation` runs block the item's schedule** ŌĆö the engine
  does not call `_advance` until the run is resolved (approved / denied /
  timeout). This is intentional: a cron-driven `act` item shouldn't fire a
  second plan while the first is still parked.
- **Token secrecy.** Tokens never appear in `GET /api/autonomy/runs` or in
  the live WS feed. The notification is the only path that carries the
  token, so the confirmation channel must be the one you trust to reach
  you.
- **Partial-success `act` runs.** With `strict_execute=false` (default), a
  mid-plan tool failure still marks the overall run `ok` ŌĆö check the
  `action_audit` for the per-step `executed`/`error` breakdown. Set
  `strict_execute=true` to force `status='error'` whenever any step fails.
- **Music Assistant version.** `mass_play_announcement` relies on
  `music_assistant_client.Players.play_announcement`; the current signature
  takes `volume_level: int | None` (0ŌĆō100). The wrapper accepts 0.0ŌĆō1.0
  floats too and normalises.
